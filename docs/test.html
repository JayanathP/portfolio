Documentation Guidelines for openAPI

Title and Overview

  • Specify the title in the OpenAPI 3 summary. Start with a verb, use title case, and keep it to 40 or fewer characters.
  • Provide an overview of the endpoint in the description tag in the Path Object. Explain what the endpoint does, and highlight any special behaviors. Use RST markup, multiple paragraphs, enumerated lists, tables, links, and other formatting as needed to add clarity.

Parameters and Attributes

  • Parameters are the changeable values for the URI, Header, and Query String. Attributes are the changeable values in the request and response bodies.
  • Since parameters and attributes can change, the changes are significant. The most important piece of information for each is "Why change it?"
  • Provide the default value, the expected range of values, and describe the tradeoffs when changing the value.
  • When defining boolean parameters, explicitly state what happens when the value is true and what happens when the value is false, even if it seems obvious to you.
  • Always name boolean parameters in the affirmative. For example, rather than using isDisabled, use isEnabled, then set it to false. Otherwise, the logic becomes "if true, do not do this" and the even more confusing "if false, do this."
  • Give parameters and attributes short, unambiguous names. When introducing a new term, check the Alex glossary to verify that the term isn't already in use.
  • Watch out for overloaded terms like item, widget, element, object when describing components of your API (watch out for component, too). It's better be redundant than try to find creative ways of talking about the same thing. Always choose clarity over novelty. If you choose to use an overloaded term, use it consistently throughout your API â€” don't call it a widget in one place and an element in another.

For more guidance on how to document specific parameter types, see Guidelines for Documenting Endpoints in OpenAPI 3 Files.

Populating Examples

The cURL request/response example in the API reference is constructed by pulling example values of all parameters together. For maximum clarity and illustration:

  • Specify a value for each parameter in the endpoint.
  • Specify values that are part of the same cURL request/response.
  • Specify real world values, rather than placeholders.
  • Use the example (sometimes x-example) tag to specify example values for all types of parameters.

Guidelines for Documenting Endpoints in OpenAPI 3 Files

This section is intended to clarify for feature teams the elements technical writers and other reviewers are looking for when we receive your OpenAPI 3 file and create final documentation. We hope the suggestions on this page make the creation of your OpenAPI 3 file reference easier, or at least less confusing. The items described here are not expected to be present for the initial documentation review, but they are welcome as early as possible. The final section, "How do I?", answers questions we are often asked about special publishing features you can use.

Describing the Endpoint

Guidelines for Page Title

  • Title is specified via the OpenAPI 3 summary tag in the Path Object.
  • Keep to fewer than 40 characters (Ideal fewer than 25 characters).
  • Use Title Case
  • Typically, start with a verb.

Guidelines for Endpoint Overview

  • Content is specified via the OpenAPI 3 description tag in the Path Object.
  • Specify:
    • What does this endpoint do?
    • Any special behaviors?
      (For example, the Model Derivative POST job endpoint creates a new manifest only when you translate a source design for the first time. When you translate the source design again, it updates the same manifest rather than create a new manifest, even if you are translating to a file format that the source file was not translated to previously.)
  • Include as much detail as required.
  • If necessary, you can use:
    • Multiple paragraphs, as inserted by Stoplight.
    • Inline RST markup such as bold, italics, hyperlinks, and inline literals.
    • RST bulleted and numbered lists.
    • RST tables.

Describing Parameters

Guidelines Applicable to All Parameter Types

  • Content is specified via the OpenAPI 3 description tag in Parameter Objects.
  • Specify:
    • What is this?
    • What is its significance?
  • If the parameter is not required, specify:
    • Does it have a default value, and if so what is it?
    • Are there any conditions that make this a required parameter?
  • If necessary, you can use:
    • Multiple paragraphs, as inserted by Stoplight.
    • Inline RST markup such as bold, italics, hyperlinks, and inline literals.
    • RST bulleted numbered lists.
    • RST simple tables.
  • Do not use RST code-block directives.

Guidelines Applicable to Specific Parameter Types

Boolean Parameters

  • Specify:
    • What does it mean for the option to be true?
    • What does it mean for the option to be false (specify this even if it is obvious from the description of the "true" state)?
  • Use the format "`` true `` : What happens when true ."

  • If the parameter is optional, append (Default) to the beginning of the description.

  • Always name boolean parameters in the affirmative. For example, do not use isDisabled. Use isEnabled, then set the value to false if that is the default. Otherwise, the logic, in practice, becomes "true, do not do this" versus the more confusing "false, do this."

Specifying Example Values

The cURL request/response example in the API reference is constructed by pulling example values of all parameters together. Hence:

  • Specifying a value is mandatory
  • The example values you specify for all parameters must be part of the same cURL request/response.
  • For OpenAPI 3, use the example tag to specify example values for all types of parameters.
  • For Swagger 2.0:
    • Use the example tag to specify example values for Body Parameters.
    • Use the x-example tag to specify example values for header, URI, and Query parameters.

To enable this feature, add an x-use-rst-examples : true tag to the swagger file. If you add the tag directly under the info tag, all endpoints take their examples from RST files. If you add the tag directly under an operation, only that endpoint is impacted.

Note: A similar tag named x-add-rst-examples: true is used by Forge Data Management. The only difference between x-use-rst-example and x-add-rst-example is that x-add-rst-example : true does not remove any example values that are defined via the example tag. This is useful when you want to provide more than one example for an endpoint that already has its example values specified via example tags

Exclude a Parameter from the Request/Response Example

Some parameters don't work together with other parameters, and must be excluded from the request/response example.

In such a case,

  1. Add an x-use-rst-examples : true tag as a child of the operation. The endpoint tagged in this manner will not generate examples automatically. Instead, it will insert an RST include directive in the position it would otherwise have inserted an example.
  2. Take note of the RST file that the include directive refers to, and create that RST file. This RST file is expected to contain a manually written example.
  3. During the build process, the endpoint RST and the example RST will get combined together and rendered as a single endpoint API reference topic.
